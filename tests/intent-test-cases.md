# AI 聊天意图识别 — 完整边界测试用例（v2.0）

**版本**：v2.0（含 2026-07-18 用户原始场景）
**目标覆盖率**：100%
**文档目的**：穷举用户可能输入的不同表达，确保 AI 正确执行意图 + 仪表盘/日程表/知识库同步可验证

---

## 命名规范

- `IT-XXX` 基础意图测试（Intent Test）
- `MIT-XXX` 多意图组合测试（Multi-Intent Test）— **核心场景：标题必须简化、必须拆解**
- `EIT-XXX` 边缘/无效输入测试
- `RDT-XXX` 相对日期解析测试
- `EDT-XXX` 边界场景

每个用例字段：
- **场景**：意图描述
- **用户输入**：5-8 个不同表述
- **期望意图**：IntentType
- **期望标题**：≤15 字总结（**不再保留原话**）
- **期望实体**：entities
- **验证方式**：API + 仪表盘验证
- **验证项**：哪个模块能看到

---

# 🚨 第一部分：标题必须总结（关键回归）

> **核心规则**：用户原话"明天上午十点有个会，记到todo里" 不应该是保存到数据库的 title。
> AI 必须提取核心动作并简化。验证方式：API 返回的 entities.title 不能包含"记到todo里"。

## IT-001 标题清洗 - 单动词
**场景**：用户用完整句子发布指令
**用户输入变体**：
1. "帮我记一下明天上午有个会" → 期望 title: "明天上午会议"
2. "请帮我加一个任务：写文档" → 期望 title: "写文档"
3. "我想记一个术语叫 K8s" → 期望 title: "K8s"
4. "帮我把客户调研转给张工" → 期望 title: "客户调研"

**期望意图**：根据内容
**验证项**：
- 数据库中 entities.title 长度 ≤ 30 字
- 不含"帮我/请/烦请/要"等口语化引导词
- 不含"记到/加到/放入"等动作指令

---

# 🚨 第二部分：复合消息必须拆解（用户原始场景）

## MIT-001 — 用户的原话（最高优先级）
**场景**：用户说"明天上午十点有个会，记到todo里"
**用户输入**：
1. "明天上午十点有个会，记到todo里"
2. "明天下午2点开个会，待办里记一下"
3. "明天10点有个客户演示，也加个todo"
4. "今天下午3点有个会议，记得写进todo"

**期望意图**：**[ADD_MEETING, ADD_TODO]**（**必须 2 个 intents**）

**期望实体（ADD_MEETING）**：
```json
{
  "title": "明天10点会议",
  "dueDate": "<明天 10:00:00 ISO>"
}
```

**期望实体（ADD_TODO）**：
```json
{
  "title": "为明天10点会议做准备",
  "dueDate": "<明天 10:00:00 ISO>"
}
```

**验证方式**：POST /api/ai/chat
**验证项**：
- ✅ API 返回 `intents.length >= 2`，含 ADD_MEETING 和 ADD_TODO
- ✅ 仪表盘"待跟进"卡片显示新创建的 todo（**不是 meting**）
- ✅ 日程表日历在明天对应日期显示任务徽章
- ✅ 数据库中：1 条 Card(type=MEETING) + 1 条 Task(type=TODO)

---

## MIT-002 任务 + 委托
**场景**：用户同时表达自己要做的事 + 委托
**用户输入**：
1. "王工负责排查登录报错，晚上8点追问；同时我下午也得看看日志"
2. "让张工调研竞品，明早10点催；我自己也得读一下报告"

**期望意图**：[ADD_DELEGATION, ADD_TODO]
**验证项**：
- 仪表盘"待跟进"显示 1 条委托 + 1 条 TODO
- 数据库 1 条 Delegation + 1 条 Task(type=TODO)

**用户输入**：

**期望意图**：[ADD_TERM, ADD_TERM, ADD_TERM]
**验证项**：数据库创建 3 条 Card(type=TERM)

**场景**：用户说"明天 4 个 task"

**期望意图**：[ADD_TODO × N]

# 第三部分：8 种意图 × 5-8 种表述

## 1.1 ADD_TERM

### IT-010 记 K8s（基础场景）
**用户输入变体**：
1. 记一下 K8s 不太懂
2. K8s 不太懂，记一下
3. 帮我记一下 K8s
4. log K8s, 我不太懂  ← **英文混合**
5. 记个 K8s
6. jot down K8s   ← **纯英文**
7. K8s 是个新词，待了解
8. 术语：K8s

**期望**：title = "K8s"
**验证项**：GET /api/knowledge?type=TERM 列表新增 "K8s"

---

### IT-011 记术语带上下文
**用户输入**：
1. 记一下 K8s，它是容器编排工具，明天再细看

3. 帮我记下来：xxl-job 是分布式任务调度框架
4. 哦对，Vector DB 矢量数据库，记一下
5. PII（个人识别信息）这个缩写我不懂，记一下

**期望**：title = 术语缩写

## 1.3 ADD_MEETING

### IT-030 记会议（**用户原始场景变体**）
**用户输入变体**：
1. 记会议：今天下午 3 点和王工对一下登录页方案
2. 帮我记录会议要点：需求评审通过
3. 会议：周三产品评审通过，确定 V2 上线 7/25
4. 记一下：今天的 standup 没人提到延期风险
5. meeting note：Q3 OKR 已对齐
6. **明天下午 3 点有个会**    ← 关键词 fallback 必须识别为 MEETING

**期望**：
- title ≤ 20 字总结（如"下午3点和王工对登录页方案"）
- dueDate 包含日期时间

# 第四部分：相对日期解析

### RDT-001 今天相关
**输入**："今天下午 3 点"、"今晚 8 点"、"中午 12 点"
**期望**：日期=今天，时间=对应 HH:00

### RDT-002 明天相关（用户原始场景）
**输入**：
1. "明天下午 3 点"
2. "明天上午 10 点"   ← **核心场景**
3. "明早 9 点"
4. "明天晚上 8 点"

**期望**：日期=今天+1，时间=对应 HH:00

### RDT-005 eod / 下班前
**输入**："eod"、"今天下班前"
**期望**：今天 18:00

### RDT-012 跨年/季节
**输入**："Q3"、"年底"
**期望**：Q3 最后一天 23:59

# 第七部分：用户视角验收（最重要）

> 以下场景必须在 Playwright 实测中通过：

## AC-01 移动端语音完整流程
1. 打开 `/chat` 页
2. 长按"按住 说话"按钮
3. 说话（"明天下午3点有个会"）
4. 松开 → 按钮变绿 → 自动发送
5. 仪表盘/日程表可见

**验证项**：
- 录音中按钮变红脉动 ✓
- 松开后 toast 显示"已识别：XXX" ✓
- 自动跳到 chat 页并发送 ✓
- 失败时显示错误 toast ✓

## AC-02 用户原始场景端到端
1. 在 chat 页输入 "明天上午十点有个会，记到todo里"
2. 点击发送
3. 等 3 秒

**验证项**（**端到端**）：
- API 返回至少 2 个 intents（MEETING + TODO）
- 仪表盘"待跟进"面板有 1 条 TODO
- 日程表明天有 1 条会议 + 1 条 TODO
- 标题是简化版，不是原话
- 数据库 Task.type = "TODO"

## AC-03 历史持久化
1. 在 chat 页发 3 条消息
2. 刷新页面
3. 历史完整保留
4. 有"清空"按钮

## AC-04 错误处理
1. 拒绝麦克风权限
2. 显示 toast: "麦克风权限被拒绝"
3. 自动 fallback 到打字输入
